## Preface

I had the idea of implementing my own [arbitrary precision arithmetic](https://en.wikipedia.org/wiki/Arbitrary-precision_arithmetic) library a while back when I was trying to calculate $\pi$ on my computer. Back then, I was doing it with Python, which already had [language native big integer](https://peps.python.org/pep-0237/) implementation. However, it did not have native big floating point implementations. Since $\pi$ is a floating point number, I needed some external library that provided such functionality. 

I was also trying learn C++ at the time, I thought that instead of downloading a pre-built library, I would take the challenge of trying to implement the arbitrary precision floating point numbers myself. Currently, I'm still a far way off from finishing the project, so this is going to be a devlog-ish post that I'm writing. 

## Initial idea

I decided to implement big integers first. My approach for big integers was that since all numbers in computers are stored in binary formats, I would just store my number in a dynamic array of boolean. That way I can resize it to any size to represent any number, without being limited to 32-bits or 64-bits. 

```cpp
class BigInt {
public:
    BigInt(int64_t x) { ... }
    BigInt operator+(const BigInt& rhs) const { ... }
    BigInt operator-(const BigInt& rhs) const { ... }
    std::string to_string() const { ... }

    // More utility functions

private:
    bool m_positive = true;
    std::vector<bool> m_storage = { };
};
```

Then, I tested to check the performance of my implementation, and I found out that it is really slow. For example, I tried to use repeated addition to calculate $2^{65536}$ with the following code:

```cpp
int main() {
    BigInt x(1);
    for (int i = 0; i < 65536; i++)
        x = x + x;
    std::cout << x.to_string() << '\n';
    return 0;
}
```

My implementation chugged along for about 20 seconds before spewing out a number. In comparison, Python's integer implementation calculated the number in about 150 ms. That is a **huge** performance difference; clearly there are some work to do before my algorithm can go on to calculate $\pi$. 

To try to improve the performance, I speculated that any of the following could be the case that's causing my implementation's performance to be awful:

 - My addition algorithm is inefficient. 
 - The Python script was doing some funky optimization by recognizing that I'm trying to compute some integer power and using its fast power algorithm instead.
 - The `std::vector` class was storing the booleans in a weird way where each element takes up more than one byte and is therefore causing more computations. 
 
I quickly disproved these speculations:

 - My addition algorithm is already $O(n)$, and I didn't think that could be optimized further.
 - The addition-to-power thing sounds a bit too advanced for the Python interpreter to detect. Besides, I had the interpreter optimization off anyway. 
 - The C++ standard doesn't specify booleans to be one bit, but it does specify `std::vector<bool>` to be optimized to a vector containing one bit booleans. 

## Python's Big Integers

Without further leads as to why my implementation is so slow, I turned to the internet for clues. It turns out, Python's internal representation of the big integers [is in base $2^{30}$](https://rushter.com/blog/python-integer-implementation/). This means each digit can represent an integer from $0$ to $2^{30} - 1$. 

I was a bit confused at first since it's not normal for one to use a billion as a base in any number system. However, it made perfect sense because the higher the base is, the larger the number each of the elements of the dynamic array can store, which means there will be less elements in the array. For example, for a 96-bit integer, my boolean implementation needs 96 elements in the dynamic array, while Python's implementation only needs 3 elements. 

Since the CPU's [built-in ALU](https://en.wikipedia.org/wiki/Arithmetic_logic_unit) is **much faster** at operations than us manually looping through an array of booleans, it makes sense for each element of the dynamic array to represent as much bits as possible. In terms of why Python's implementation chose base $2^{30}$ as opposed to $2^{32}$, it is probably implementation details.

I rewrote my code using 32-bit "digits" in the storage array:

```cpp {hl_lines=[12]}
class BigInt {
public:
    BigInt(int64_t x) { ... }
    BigInt operator+(const BigInt& rhs) const { ... }
    BigInt operator-(const BigInt& rhs) const { ... }
    std::string to_string() const { ... }
    
    // More utility functions

private:
    bool m_positive = true;
    std::vector<uint32_t> m_storage = { }; // uint32_t's this time
};
```

When I ran the speed test again, my implementation of addition performed than Python's. The speed increase likely came from using 2 more bits per "digit". 

However, after computing the correct value for $2^{65536}$, my program took about 5 extra seconds of time to convert from the storage base to base-10 for display. In contrast, Python took merely 20ms this time. This obviously wasn't going to scale well. Since being able to print the value of $\pi$ out after computation is important, we also need to improve this algorithm. 

This will be discussed in the next blog posts.

## Code

Here are the final internal implementation of integer addition and subtraction for big integers in represented in base $2^{32}$. Note that these functions do not handle the sign of the integer. You can also [view the full project on GitHub](https://github.com/somedude72/vint?tab=readme-ov-file).

```cpp {linenos=inline}
using storage_t = std::vector<uint32_t>;

inline storage_t add_int(const storage_t& a, const storage_t& b) {
    const uint64_t BASE = (1ULL << 32);
    const size_t n = std::max(a.size(), b.size());
    
    uint64_t carry = 0;
    storage_t result(n);

    for (size_t i = 0; i < n; i++) {
        const uint64_t ai = (i < a.size()) ? a[i] : 0;
        const uint64_t bi = (i < b.size()) ? b[i] : 0;
        uint64_t sum = ai + bi + carry;
        result[i] = static_cast<uint32_t>(sum & (BASE - 1));
        carry = (sum >= BASE);
    }

    if (carry) result.push_back(1);
    return result;
}

inline storage_t sub_int(const storage_t& a, const storage_t& b) {
    const uint64_t BASE = (1ULL << 32);
    const size_t n = a.size();

    uint64_t borrow = 0;
    storage_t result(n);

    for (size_t i = 0; i < n; i++) {
        uint64_t ai = (i < a.size()) ? a[i] : 0;
        uint64_t bi = (i < b.size()) ? b[i] : 0;
        uint64_t diff = ai - bi - borrow;
        result[i] = static_cast<uint32_t>(diff & (BASE - 1));
        borrow = (ai < (bi + borrow));
    }

    while (!result.empty() && result.back() == 0) result.pop_back();
    return result;
}
```
