## Preface

I had the idea of implementing my own [arbitrary precision arithmetic](https://en.wikipedia.org/wiki/Arbitrary-precision_arithmetic) library a while back when I was trying to calculate $\pi$ on my computer. Back then, I was doing it with Python, which already had [language native big integer](https://peps.python.org/pep-0237/) implementation. However, it did not have native arbitrary precision floating point implementations, and since $\pi$ is not an integer, I was forced to use some external library that provided such functionality. 

Because I was trying to also learn C++ at the time, I thought that I would take the challenge of trying to implement the arbitrary precision floating point numbers myself instead of downloading a pre-built library. I guess this is somewhat akin to this guy [writing his own memory allocator](https://www.youtube.com/watch?v=sZ8GJ1TiMdk), except I have much less experience; but hey, that's where all the fun lies.

Currently, I'm still a far way off from finishing the project, so this is going to be a devlog-ish post that I'm writing. However, just between you and me, the actual reason I'm writing this blog post is because I just discovered that Hugo [has $\LaTeX$ rendering capabilities](https://gohugo.io/content-management/mathematics/), and I really wanted to play around with it. 

## Representing the Bits

My initial plan was that since all numbers in computers are stored in binary formats, I would just store my number in a dynamic array of boolean. That way I can resize it to any size to represent any number, without being limited to 32-bits or 64-bits. I promptly got to work and implementing the algorithms for the big integer. 

However, when I tested them to check their performance, I found out that they were really slow. For example, I tried to use repeated addition to calculate $2^{65536}$ with the following code (yeah, it's a bit of an arbitrary number):

```cpp
class BigInt {
public:
    BigInt(int64_t x) { ... }
    BigInt operator+(const BigInt& rhs) const { ... }
    BigInt operator-(const BigInt& rhs) const { ... }
    std::string to_string() const { ... }
private:
    std::vector<bool> m_storage = { };
};

int main() {
    BigInt x = 1;
    for (int i = 0; i < 65536; i++)
        x = x + x;
    std::cout << x.to_string() << '\n';
    return 0;
}
```

This program chugged along for about 30 seconds before spewing out a number. In comparison, when I ran the exact same script with Python's integer implementation, I timed it and found out that it calculated the number in about 100 ms. That is a huge performance difference; clearly there are some work to do before my algorithm can go on to calculate $\pi$. 

To try to improve the performance, I speculated that any of the following could be the case that's causing my implementation's performance to be awful:

 - My addition algorithm is inefficient. 
 - The Python script was doing some funky optimization by recognizing that I'm trying to compute some power and using its fast power algorithm instead.
 - The `std::vector` class was storing the booleans in a weird way where each element takes up more than one byte and is therefore causing more computations. 
 
I quickly disproved these speculations:

 - My addition algorithm is already $O(n)$, and I didn't think that could be optimized further.
 - The addition-to-power thing sounds a bit too advanced for the Python interpreter to detect. Besides, I had the interpreter optimization off anyway. 
 - The C++ standard doesn't specify booleans to be one bit, but it does specify `std::vector<bool>` to be optimized to a vector containing one bit booleans. 

Without further leads as to why my implementation is so slow, I turned to the internet for the search. It turns out, Python's internal representation of the big integers [is in base $2^{30} - 1$](https://rushter.com/blog/python-integer-implementation/).
